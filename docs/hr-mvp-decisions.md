# HR MVP 운영 규칙 정의서

## 1. 문서 목적

이 문서는 HR 입퇴사/조직도 조회 MVP에서 사용하는 운영 규칙과 계산 기준을 고정하기 위한 문서다.

- 기준일 조회(`as-of date query`) 시 숫자가 흔들리지 않도록 한다.
- 시트 입력 구조와 계산 로직의 기준 문서로 사용한다.
- 이후 규칙이 바뀌면 이 문서를 먼저 수정한 뒤 구현을 변경한다.

## 2. 문서 상태

- 문서 버전: `v0.1`
- 작성일: `2026-04-30`
- 상태: `초안이지만 현재 합의사항 반영`

## 3. 확정된 운영 규칙

| 항목 | 결정 |
|---|---|
| 기준일 조회 방식 | 특정 날짜 `D` 기준 스냅샷으로 조회 |
| 주 소속 팀(primary team) | 한 시점에 1명당 1개만 허용 |
| 팀장 수 | 한 팀에 같은 시점 팀장 1명만 허용 |
| 팀장 공석 | 허용 |
| 휴직 중 팀 이동 | 허용 |
| 퇴사일 처리 | 퇴사일 당일까지 `재직`으로 간주 |
| 팀장 데이터 성격 | `팀 이동`이 아니라 `역할(role assignment)` |
| 팀 계층 변경기록 의미 | 팀의 상위 조직, 활성 상태, 팀명 이력 관리 |

## 4. 아직 미정인 항목

### 4.1 팀명 변경 처리 방식

현재 미정이다.

선택지는 보통 2가지다.

1. `같은 팀 ID 유지 + 팀명만 변경`
2. `새 팀으로 간주하여 새 team_id 생성`

권장안:

- 조직의 실체가 같고 이름만 바뀐 경우:
  - `같은 team_id 유지`
  - `Team_Structure_History`에서 기간별 `team_name` 변경
- 책임 범위, 리더십, 예산 단위, 공식 조직 코드가 바뀌어 별도 조직으로 보는 편이 맞는 경우:
  - `새 team_id 생성`

즉 `이름 변경(rename)`과 `조직 재편(reorganization)`을 구분하는 운영 원칙이 필요하다.

## 5. 상태 축 분리 원칙

이 MVP는 인사 데이터를 아래 4개 축으로 나눠서 관리한다.

| 상태 축 | 의미 | 권장 시트 |
|---|---|---|
| 고용 상태 | 회사 소속 여부 | `Employment_Periods` |
| 근무 예외 상태 | 휴직 여부 | `Leave_Periods` |
| 소속 상태 | 어느 팀 소속인가 | `Team_Assignments` |
| 역할 상태 | 팀장/본부장 등 보직 | `Role_Assignments` |
| 조직 구조 | 팀의 상하위 관계 | `Team_Structure_History` |

핵심 원칙:

- `입사/퇴사`와 `팀 이동`은 다르다.
- `팀 이동`과 `팀장 임명`도 다르다.
- `팀장 임명`과 `팀 계층 변경`도 다르다.

## 6. 기준일 계산 규칙

기준일을 `D`라고 할 때 아래 규칙으로 계산한다.

### 6.1 재직 여부

아래 조건을 만족하면 `재직`이다.

- `Employment_Periods.employment_start <= D`
- 그리고 `employment_end`가 비어 있거나 `D <= employment_end`

보충 규칙:

- `employment_end = D`인 경우에도 재직으로 본다.

### 6.2 휴직 여부

아래 조건을 만족하면 `휴직`이다.

- `Leave_Periods.leave_start <= D`
- 그리고 `leave_end`가 비어 있거나 `D <= leave_end`

보충 규칙:

- 휴직자는 `비재직`이 아니라 `재직 중 비근무` 상태다.

### 6.3 소속 팀 판단

아래 조건을 만족하는 `Team_Assignments` 행 중
`is_primary = TRUE`인 1개 행으로 주 소속 팀을 판단한다.

- `assignment_start <= D`
- 그리고 `assignment_end`가 비어 있거나 `D <= assignment_end`

검증 규칙:

- 같은 날짜에 `is_primary = TRUE` 행이 2개 이상이면 데이터 오류다.

### 6.4 팀장 판단

아래 조건을 만족하는 `Role_Assignments` 행으로 팀장을 판단한다.

- `role_code = TEAM_LEAD`
- `role_start <= D`
- 그리고 `role_end`가 비어 있거나 `D <= role_end`

검증 규칙:

- 같은 팀(`scope_team_id`)에 같은 날짜 유효한 `TEAM_LEAD` 행은 최대 1개만 허용한다.
- 해당 행이 없으면 팀장 공석으로 본다.

### 6.5 조직도 트리 판단

조직도 트리는 `Team_Structure_History`의 기간 유효 행으로 계산한다.

- `effective_from <= D`
- 그리고 `effective_to`가 비어 있거나 `D <= effective_to`

조직도 표현은 아래 두 레이어를 합쳐서 만든다.

1. 팀 트리
2. 각 팀의 구성원과 팀장

## 7. 시트별 책임 범위

| 시트명 | 무엇을 저장하는가 | 저장하면 안 되는 것 |
|---|---|---|
| `Employees` | 직원 마스터 | 날짜별 상태 이력 |
| `Employment_Periods` | 입사, 퇴사, 재입사 | 팀 이동, 팀장 정보 |
| `Leave_Periods` | 휴직 기간 | 퇴사 처리 |
| `Team_Assignments` | 팀 소속 기간 | 팀장 여부 |
| `Role_Assignments` | 팀장 등 보직 기간 | 조직 상하위 관계 |
| `Team_Structure_History` | 팀의 부모 조직, 팀명 이력 | 사람 배치 정보 |

## 8. 필수 검증 규칙

MVP에서 최소한 아래 검증은 자동으로 잡아야 한다.

1. 한 사람의 `Employment_Periods`가 서로 겹치면 오류
2. 한 사람의 주 소속 팀(`is_primary = TRUE`) 기간이 겹치면 오류
3. 한 팀의 `TEAM_LEAD` 기간이 겹치면 오류
4. 종료일이 시작일보다 빠르면 오류
5. 재직하지 않는 기간에 팀 소속이 존재하면 경고 또는 오류
6. 재직하지 않는 기간에 팀장 역할이 존재하면 오류
7. 존재하지 않는 `employee_id`, `team_id` 참조 시 오류

## 9. 조회 지표 정의

| 지표 | 정의 |
|---|---|
| 총 재직 인원 | 기준일 `D`에 재직인 사람 수 |
| 총 휴직 인원 | 기준일 `D`에 휴직인 사람 수 |
| 총 근무 인원 | 총 재직 인원 - 총 휴직 인원 |
| 팀별 재직 인원 | 기준일 `D`에 해당 팀 주 소속인 재직자 수 |
| 팀별 휴직 인원 | 기준일 `D`에 해당 팀 주 소속이면서 휴직인 사람 수 |
| 팀별 근무 인원 | 팀별 재직 인원 - 팀별 휴직 인원 |
| 당해연도 퇴사자 수 | 기준일이 속한 연도의 `1월 1일 ~ D` 사이 `employment_end` 건수 |
