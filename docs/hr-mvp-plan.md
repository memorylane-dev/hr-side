# HR 입퇴사/조직도 조회 MVP 요구사항 초안

## 1. 목표

직원 100명 내외 규모의 회사에서 2015년부터 현재까지의 인사 이력 데이터를 바탕으로,
특정 날짜 기준의 조직도와 인사 통계를 조회할 수 있는 MVP를 만든다.

핵심 질문은 아래 3가지다.

1. `YYYY-MM-DD` 시점에 누가 재직 중이었는가?
2. 그 시점에 각 직원은 어느 팀에 속해 있었는가?
3. 해당 연도 기준으로 입사/퇴사/휴직 통계는 어떻게 되는가?

## 2. MVP에서 반드시 풀어야 할 문제

조직도를 과거 시점으로 복원하려면 현재 상태만으로는 부족하다.
아래 이력이 최소한 있어야 한다.

- 직원 기본 정보
- 입사일
- 퇴사일 또는 퇴사 이벤트
- 휴직 시작일/종료일
- 팀 이동 이력
- 팀의 상위 조직 변경 이력

특히 `팀 이동 기록`만 있고 `팀 계층 변경 기록`이 없으면 과거 조직도를 정확히 복원할 수 없다.

## 3. MVP 사용자 시나리오

사용자는 Google Spreadsheet에 인사 이력을 입력하거나 CSV로 붙여넣는다.
조회 화면에서 기준 날짜를 선택하고 `스냅샷 생성` 버튼을 누른다.
그러면 아래 결과가 나온다.

- 해당 날짜 기준 조직도
- 팀별 재직 인원 수
- 팀별 근무 인원 수
- 팀별 휴직 인원 수
- 기준 연도 입사자 수
- 기준 연도 퇴사자 수
- 기준 연도 순증감

## 4. 추천 구현 방향

### 4.1 1차 MVP 권장안: Google Sheets + Apps Script

이 규모에서는 이 방식이 가장 현실적이다.

- 장점
  - HR 담당자가 직접 데이터 입력/수정 가능
  - 별도 배포 없이 빠르게 MVP 제작 가능
  - 직원 100명 수준에서는 성능 부담이 거의 없음
  - 향후 웹 앱으로 옮길 때 데이터 구조를 재사용 가능

- 단점
  - 조직도 UI가 웹보다 제한적임
  - 권한 관리, 감사 로그, 고급 필터는 약함
  - 여러 사용자가 동시에 복잡한 편집을 하면 운영 규칙이 필요함

### 4.2 2차 확장안: 웹 앱

아래 조건이 생기면 웹 전환을 고려한다.

- 조회 사용자가 많아짐
- 권한 관리가 필요함
- 부서장/HR/경영진별 뷰가 달라야 함
- 시각화 품질을 높이고 싶음
- ERP/HRIS 연동이 필요함

권장 웹 스택 예시는 아래와 같다.

- 프론트엔드: Next.js
- 백엔드/DB: Supabase 또는 PostgreSQL
- 차트/조직도: React Flow 또는 D3 계열

## 5. 데이터 모델 제안

MVP에서는 `현재 상태 테이블`보다 `기간(period) 기반 이력 테이블`이 중요하다.

이번 요구사항에서는 상태를 한 덩어리로 보지 말고 아래 4개 축으로 분리하는 것이 맞다.

1. `고용 상태(employment lifecycle)`
2. `소속 상태(team assignment)`
3. `역할 상태(role assignment)`
4. `조직 구조(team hierarchy)`

즉 `입사/퇴사`, `팀 이동`, `팀장 임명/해제`, `팀의 상위 조직 변경`은 서로 다른 종류의 변화다.
같은 `이벤트`처럼 보여도 계산 규칙이 다르므로 분리 저장하는 편이 낫다.

특히 아래 구분이 중요하다.

- `팀장 임명/해제`는 `팀 계층 변경`이 아니다.
- `팀장 임명/해제`는 보통 `역할(role)` 변경이다.
- `팀 이동`은 `소속(team membership)` 변경이다.
- `팀 계층 변경`은 `조직도 트리(parent-child relation)` 변경이다.

스프레드시트 기반 MVP에서는 `모든 것을 Employee_Events 한 시트에 넣는 방식`보다
`도메인별 기간 테이블`이 입력도 쉽고 조회 로직도 단순하다.

### 5.1 `Employees` 시트

직원 마스터 정보.

권장 컬럼:

- `employee_id`
- `name`
- `email` (선택)
- `employment_type` (정규직/계약직 등, 선택)
- `initial_hire_date`
- `final_termination_date` (선택, 최종 상태 참고용)

주의:

- 이름이 아니라 `employee_id`를 기준키로 사용한다.
- 퇴사 후 재입사 가능성이 있으면 아래 `Employment_Periods`가 진실원장(source of truth)이 된다.

### 5.2 `Teams` 시트

팀 마스터 정보.

권장 컬럼:

- `team_id`
- `team_name`
- `team_type` (본부/실/팀 등, 선택)
- `active_from`
- `active_to`

### 5.3 `Employment_Periods` 시트

입사/퇴사/재입사를 표현하는 고용 기간 이력.

권장 컬럼:

- `employment_period_id`
- `employee_id`
- `employment_start`
- `employment_end`
- `hire_reason` (신규입사/재입사 등, 선택)
- `termination_reason` (선택)
- `note`

설명:

- 한 사람이 퇴사 후 다시 입사하면 행이 2개 이상 생길 수 있다.
- 이 시트는 `그 날짜에 회사 소속인가`를 판단하는 기준이다.
- `팀 이동`이나 `팀장 임명`은 여기 넣지 않는다.

### 5.4 `Leave_Periods` 시트

휴직 기간 이력.

권장 컬럼:

- `leave_id`
- `employee_id`
- `leave_type`
- `leave_start`
- `leave_end`
- `note`

설명:

- 휴직은 `퇴사`가 아니라 `재직 중 비근무 상태`다.
- 따라서 고용 이력과 분리하는 것이 맞다.

### 5.5 `Team_Assignments` 시트

직원의 팀 소속 기간 이력.

권장 컬럼:

- `assignment_id`
- `employee_id`
- `team_id`
- `assignment_start`
- `assignment_end`
- `is_primary`
- `assignment_type` (정식배치/겸직/파견 등, 선택)
- `note`

설명:

- `팀 이동`은 이전 행 종료 + 새 행 시작으로 표현한다.
- 한 시점에 1인 1주팀만 허용할지, 겸직을 허용할지 초기에 정해야 한다.
- MVP는 `is_primary = TRUE`인 주 소속 1개만 허용하는 것을 권장한다.

### 5.6 `Role_Assignments` 시트

직원의 역할/직책/보직 기간 이력.

권장 컬럼:

- `role_assignment_id`
- `employee_id`
- `role_code`
- `scope_type`
- `scope_team_id`
- `role_start`
- `role_end`
- `is_acting`
- `note`

`role_code` 예시:

- `TEAM_LEAD`
- `HEAD`
- `DIRECTOR`

설명:

- `팀장 임명/해제`는 여기에 넣는다.
- 같은 팀 소속을 유지한 채 `팀원 -> 팀장`, `팀장 -> 팀원`이 될 수 있으므로
  `Team_Assignments`와 분리해야 한다.
- `scope_team_id`는 어떤 팀의 팀장인지 나타낸다.

### 5.7 `Team_Structure_History` 시트

조직 계층 변경 이력.

권장 컬럼:

- `history_id`
- `team_id`
- `team_name`
- `parent_team_id`
- `effective_from`
- `effective_to`
- `status` (active/closed, 선택)

예:

- 2020-01-01 ~ 2022-12-31: `플랫폼팀`의 상위 조직은 `CTO`
- 2023-01-01 ~ 현재: `플랫폼팀`의 상위 조직은 `Product본부`

설명:

- 이것이 진짜 `팀 계층 변경기록(team hierarchy history)`이다.
- 팀 생성, 폐지, 상위조직 변경, 필요시 팀명 변경까지 여기서 관리한다.
- `누가 팀장인가`는 여기 넣지 않는다.

### 5.8 왜 이렇게 나누는가

예를 들어 아래 4가지 변경은 서로 독립적이다.

1. `2024-03-01` 입사
2. `2024-06-01` 플랫폼팀 배치
3. `2025-01-01 ~ 2025-03-31` 육아휴직
4. `2025-04-01` 플랫폼팀 팀장 임명

이것을 한 이벤트 시트로 뭉치면 조회 시 아래 로직이 다 꼬인다.

- 재직 여부
- 근무 여부
- 소속 팀
- 리더 여부

반대로 테이블을 나누면 기준일 `D`에 대해 아래처럼 계산하면 된다.

- `Employment_Periods`에서 재직 여부 계산
- `Leave_Periods`에서 휴직 여부 계산
- `Team_Assignments`에서 주 소속 팀 계산
- `Role_Assignments`에서 팀장 여부 계산
- `Team_Structure_History`에서 조직 트리 계산

## 6. 기준 날짜 계산 규칙

여기가 가장 중요하다. 이 규칙을 먼저 확정해야 숫자가 흔들리지 않는다.

### 6.1 재직 여부

기준일 `D`에 아래 조건이면 `재직`으로 본다.

- `Employment_Periods`에 `employment_start <= D <= employment_end`인 행이 있거나
- 종료일이 비어 있고 `employment_start <= D`인 행이 있음

권장 규칙:

- 퇴사일 당일까지 재직으로 볼지
- 퇴사일 다음날부터 비재직으로 볼지

보통은 `퇴사일 당일까지 재직`으로 두는 편이 실무에서 이해가 쉽다.

### 6.2 휴직 여부

기준일 `D`가 `Leave_Periods`의 휴직 기간에 포함되면 `휴직`으로 본다.

권장 출력은 3가지다.

- `재직 인원`
- `근무 인원` = 재직 인원 - 휴직 인원
- `휴직 인원`

즉 휴직자는 퇴사자가 아니라 `재직 중이지만 근무 제외` 상태로 분리한다.

### 6.3 소속 팀 판단

기준일 `D`에 유효한 `Team_Assignments` 행 중 `is_primary = TRUE`인 행을 기준으로 소속 팀을 결정한다.

즉 아래 우선순위를 사용한다.

1. `assignment_start <= D <= assignment_end`
2. 종료일이 비어 있으면 현재까지 유효
3. 같은 날짜에 2개 이상 있으면 데이터 오류로 간주

### 6.4 팀장/보직 판단

기준일 `D`에 유효한 `Role_Assignments` 행으로 역할을 판단한다.

예:

- `role_code = TEAM_LEAD`
- `scope_team_id = 플랫폼팀`
- `role_start <= D <= role_end`

이 경우 해당 날짜에는 `플랫폼팀 팀장`으로 본다.

즉 조직도에서 팀장 이름을 보여주려면
`Team_Structure_History`가 아니라 `Role_Assignments`를 조회해야 한다.

### 6.5 조직도 트리 판단

기준일 `D`에 유효한 `Team_Structure_History`를 기준으로 팀 트리를 만든다.

조직도는 아래 두 레이어를 합쳐서 표현한다.

1. `조직 노드`: 팀과 상하위 관계
2. `사람 노드`: 각 팀의 구성원과 팀장

### 6.6 당해연도 퇴사자 수

기준일 `D`가 속한 연도의 `1월 1일 ~ D` 사이
`Employment_Periods.employment_end`가 발생한 건수를 퇴사자 수로 정의하는 것을 권장한다.

예:

- 기준일이 `2026-04-30`이면 `2026-01-01 ~ 2026-04-30` 퇴사자 수

필요하면 아래도 추가 가능하다.

- 전년도 퇴사자 수
- 해당 연도 전체 퇴사 예정/확정 수
- 월별 퇴사 추이

## 7. MVP 조회 화면 요구사항

### 7.1 필수 입력

- 기준 날짜 1개

### 7.2 필수 출력

- 기준 날짜
- 총 재직 인원
- 총 근무 인원
- 총 휴직 인원
- 기준 연도 입사자 수
- 기준 연도 퇴사자 수
- 팀별 인원 집계
- 조직도 트리

### 7.3 조직도 표현 방식

1차 MVP에서는 복잡한 박스형 차트보다 아래 중 하나가 현실적이다.

1. 들여쓰기 기반 트리 표
2. 팀별 테이블 + 상위팀 컬럼
3. HTML Sidebar에서 간단한 Org Chart

가장 먼저는 `들여쓰기 기반 트리 표`를 추천한다.

예:

```text
CEO (1)
  Product본부 (12)
    플랫폼팀 (5)
      홍길동
      김개발
    데이터팀 (7)
  경영지원본부 (8)
    인사팀 (3)
```

## 8. Google Sheets 탭 구성 제안

### 8.1 입력 탭

- `Employees`
- `Teams`
- `Employment_Periods`
- `Leave_Periods`
- `Team_Assignments`
- `Role_Assignments`
- `Team_Structure_History`

### 8.2 출력 탭

- `Dashboard`
- `Snapshot`
- `Validation`

`Dashboard`:

- 날짜 선택 셀
- 주요 KPI 카드
- 팀별 요약표

`Snapshot`:

- 기준일 당시 직원별 상태 스냅샷
- 컬럼 예: 직원ID, 이름, 재직여부, 휴직여부, 소속팀, 팀장여부, 상위조직

`Validation`:

- 이벤트 오류 검출
- 기간 중복 오류 검출
- 중복 ID 검출
- 종료일이 시작일보다 빠른 데이터 검출
- 팀 없는 직원 검출
- 한 시점에 주 소속 팀이 2개인 직원 검출
- 재직하지 않는데 팀장 역할이 잡힌 행 검출

## 9. Apps Script MVP 기능 제안

- `buildSnapshot(targetDate)`
  - 기준일 스냅샷 생성

  - `데이터 검증`

## 11. 가장 중요한 운영 원칙

아래 4가지는 초기에 반드시 합의해야 한다.

1. `휴직자를 재직 인원에 포함할 것인가`
2. `퇴사일 당일을 재직으로 볼 것인가`
3. `팀이름 변경`과 `새 팀 생성`을 어떻게 구분할 것인가
4. `1인이 여러 팀에 동시에 속할 수 있는가`
5. `팀장`을 역할(role)로 볼 것인가, 직급(position)으로 볼 것인가
6. `한 팀에 같은 날짜에 팀장이 2명 있을 수 있는가`

이 정의가 확정되지 않으면 같은 데이터로도 숫자가 계속 달라질 수 있다.

## 12. 추천 MVP 로드맵

### Step 1. 데이터 정의

- 시트 구조 확정
- 상태 축 확정
- 계산 규칙 확정

### Step 2. 샘플 데이터 적재

- 실제 또는 가공 데이터 30~50건 입력
- 예외 케이스 포함
  - 입사 후 빠른 퇴사
  - 휴직 중 팀 이동
  - 조직 개편
  - 재입사

### Step 3. 스냅샷 계산

- 기준 날짜별 직원 상태 계산
- 팀 배정 계산
- 역할 배정 계산
- 조직 트리 계산

### Step 4. 대시보드 구성

- KPI
- 팀별 집계
- 조직도 표현

### Step 5. 검증

- 특정 날짜 3~5개를 골라 수기 검증
- HR 담당자가 숫자를 이해할 수 있는지 확인

## 13. 최종 추천

현재 조건이라면 아래가 가장 적절하다.

1. `Google Sheets + Apps Script`로 MVP 시작
2. 데이터 입력보다 `이력 구조`와 `계산 규칙`을 먼저 고정
3. 조직도는 처음부터 화려한 차트보다 `정확한 과거 복원`에 집중
4. 데이터와 규칙이 안정되면 그때 웹 앱으로 확장

핵심은 UI가 아니라 `특정 날짜 기준 상태를 일관되게 재구성할 수 있는 데이터 모델`이다.
